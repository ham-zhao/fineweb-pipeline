请对指定的 .md 文档执行全面质量检查。如果未指定文件，则检查当前对话中最近编辑的 .md 文件。

检查基于 `docs/METHODOLOGY_PATTERNS.md` 中的方法论模式库。每个检查项标注了对应的方法论编号，方便追溯和扩展。

> **扩展方式**：后续新增检查类别时，在下方追加新的 `## N. 检查名（方法论 #X）` 章节，并同步更新"输出要求"中的总数即可。

---

## 1. 指标口径检查（方法论 #1）

扫描文档中所有百分比和比率，检查：
- [ ] 每个百分比是否能回答"分子是什么？分母是什么？"
- [ ] 同一指标在不同章节的口径是否一致
- [ ] 绝对差和相对变化率是否有明确标注（"+10%" 是 0.10 的绝对差，还是原来的 10% 相对增长？）

**输出**：列出所有口径不清晰的指标，给出修复建议。

## 2. 英文缩写检查（方法论 #9）

- [ ] 用 `[A-Z]{2,}` 扫描文档中所有大写缩写，统计出现次数
- [ ] 超过 5 个缩写时，文档开头是否有术语表？
- [ ] 每个缩写是否至少有一次全写 + 中文简释？
- [ ] 正文中首次出现处是否用括号标注了全写？
- [ ] 同一缩写在文档中是否始终指同一含义？

**输出**：列出缺少全写的缩写，给出建议的全写和简释。如缺术语表，直接生成。

## 3. 文档逻辑性检查（方法论 #10）

- [ ] 每章开头是否有逻辑链声明？（"本章逻辑：A → B → C"）
- [ ] 同一概念是否只在一个地方完整定义？（检查重复内容）
- [ ] 所有数字/结论是否标注了来源（论文引用 or 推导过程）？
- [ ] 章节间的依赖关系是否清晰？

**输出**：列出逻辑链缺失的章节、重复定义的内容、无来源的结论。

## 4. 数值一致性检查

扫描文档中所有数字，检查：
- [ ] 同一指标在不同位置引用时数值是否一致（如某个保留率在摘要、正文、核对清单中是否相同）
- [ ] 衍生计算是否正确（如 N×40%×10% 的结果是否与下文引用一致）
- [ ] 范围值（如 30-50%）在不同位置是否一致

**输出**：列出数值不一致的位置，标注哪个是正确值。

## 5. 核对清单完整性检查（方法论 #8）

如果文档包含核对清单/验收表：
- [ ] 每个检查项是否有明确的口径定义？
- [ ] 每个检查项是否有预期值？
- [ ] 是否区分了预期值和红线？
- [ ] 偏差方向是否分别给出了偏高/偏低的建议？

**输出**：列出不完整的核对清单项。

## 6. 模块拆解检查（方法论 #3）

如果文档涉及多步 pipeline 或复合指标：
- [ ] 端到端指标是否拆解到各模块？
- [ ] 各模块的输入/输出/贡献是否量化？
- [ ] 串行 pipeline：各步存活率的乘积是否等于总存活率？
- [ ] 并行模块：各模块输出之和是否等于总输出？

**输出**：列出未拆解的复合指标或数学不一致的拆解。

## 7. 分模块结果校验覆盖检查（方法论 #3 延伸）

如果文档包含模块拆解 **且** 包含结果校验/核对清单，检查拆解是否在校验中被完整覆盖：

- [ ] 每个模块是否都有对应的校验项？（不能只校验整体，漏掉子模块）
- [ ] 校验项是否同时包含**百分比**和**绝对文档数**？（百分比可能掩盖量级问题）
- [ ] 串行 pipeline 是否有**内部一致性交叉校验**？（各步存活率乘积 ≈ 整体存活率，偏差 <5%）
- [ ] 并行/分桶是否有**完整性校验**？（各桶之和 = 输入总数；各模块输出之和 = 总输出）
- [ ] 每个模块是否独立设置了预期值和红线？（防止异常被整体指标"平均掉"）

**输出**：列出方案中有拆解但核对清单中缺少对应校验的模块，给出补充建议。

## 8. 需求-实现一致性检查（方法论 #11）

如果文档同时包含"设计/需求章节"和"执行计划/实现章节"：
- [ ] 设计章节中的每个处理步骤/模块，在执行计划中是否有对应的代码实现步骤？
- [ ] 每个实现步骤是否有明确的文件路径、函数名、验收标准？
- [ ] 新增的配置参数（阈值、开关、比例）是否在 config 创建/修改步骤中列出？
- [ ] 核对清单是否覆盖了所有新增步骤的输出验证？
- [ ] 2.3 验收表（或等价的全局验收标准）是否包含了新增模块的指标？

**输出**：列出设计中有但执行计划中缺失的实现步骤，给出补充建议。

---

## 输出要求

1. 按上述 8 个检查类别分别输出结果
2. 每类给出：通过项数 / 总检查项数
3. 汇总评分：

| 等级 | 标准 |
|------|------|
| A | 全部通过，无需修改 |
| B | 有小问题（<3 项），不影响理解 |
| C | 有重大问题（口径混淆/数值矛盾/逻辑断裂），需修复 |

4. **能直接修复的问题，直接执行修复，不要只列出问题。**
5. 修复完成后重新检查修复项，确认通过。
