请对指定的 .md 技术文档执行结构与写作规范检查。如果未指定文件，则检查当前对话中最近编辑的 .md 文件。

检查基于三个业界框架：
- **Diataxis**（文档四分法：Explanation / Reference / How-to / Tutorial 不混搭）
- **Google Design Doc**（Context -> Goals/Non-goals -> Design -> Execution）
- **Google Technical Writing**（scope/non-scope、前置关键信息、聚焦读者需求）

---

## 1. Diataxis 分层检查

扫描文档各章节，判断每章属于哪种文档类型，检查是否存在混搭：

| 类型 | 特征 | 典型内容 |
|------|------|---------|
| Explanation | 回答"为什么"，提供背景和原理 | 方法论设计、论文对比、架构决策理由 |
| Reference | 事实性描述，查阅用 | 指标口径定义、配置参数表、API 说明 |
| How-to | 操作步骤，解决具体问题 | 执行计划、部署步骤、Phase A-E |
| Tutorial | 引导学习，手把手教 | 入门教程、示例演练 |

检查项：
- [ ] 每章是否有明确的主类型？（不应同时是 Explanation + How-to）
- [ ] Explanation 内容（方法论原理）是否与 How-to 内容（执行步骤）分离？
- [ ] Reference 内容（指标定义）是否集中在一处定义，而非散落各章？
- [ ] 正文中的深度解释（blockquote / 旁注）是否过多？（>10 个 blockquote 需考虑收归附录）

**输出**：列出每章的类型标注，标记混搭的章节，给出拆分建议。

## 2. Google Design Doc 结构检查

检查文档是否包含以下关键章节（适用于技术设计/方案类文档）：

- [ ] **Overview / 摘要**：是否有 1 页以内的全文摘要？（问题 -> 方案 -> 预期结果）
- [ ] **Context / 背景**：是否简洁说明技术背景？（链接详情而非内联）
- [ ] **Goals & Non-goals**：是否明确了做什么和不做什么？
- [ ] **Design / 技术设计**：是否聚焦于 trade-off 和决策理由？（而非实现细节罗列）
- [ ] **Alternatives / 排除范围**：是否说明了不选的方案及原因？

**输出**：缺失的章节列表，给出补充建议。

## 3. Google Technical Writing 原则检查

### 3.1 Scope / Non-scope
- [ ] 文档是否在开头定义了范围（scope）和非范围（non-scope）？
- [ ] 非范围是否只列出"读者可能合理预期但本文不覆盖"的内容？（不列无关内容）

### 3.2 前置关键信息
- [ ] 文档前 2 屏（约 50 行）是否回答了读者最关心的问题？
- [ ] 每章开头是否有该章要点总结？（而非直接展开细节）

### 3.3 读者聚焦
- [ ] 目标读者是否明确？（本文面向谁？需要什么前置知识？）
- [ ] 是否避免了假设读者会从头到尾阅读？（能否跳读到任意章节理解？）

**输出**：逐项检查结果，标记不符合项。

## 4. 信息去重检查

- [ ] 同一指标/定义是否只在一个地方完整定义？（检查是否在多处重复定义）
- [ ] 验收标准是否只有一个 source of truth？（不应在 "预期指标" + "核对清单" + "整体验收" 三处重复）
- [ ] 引用是否正确？（如"详见 S2.2"，则 S2.2 是否确实存在该内容？）

**输出**：列出重复定义的内容，建议保留哪处、其余改为引用。

## 5. 执行计划可操作性检查

如果文档包含执行计划 / 步骤列表：

- [ ] 每步是否有明确的操作（做什么）+ 验收（怎么确认做完了）？
- [ ] 步骤之间的依赖关系是否清晰？（哪些可并行？哪些必须串行？）
- [ ] 步骤数量是否合理分组？（>10 步应分子组，便于追踪进度）
- [ ] 核对清单是否与指标体系对齐？（引用而非重新定义）

**输出**：列出缺少验收条件的步骤、未分组的长列表。

## 6. 版本与变更管理检查

- [ ] 文档是否标注了版本号或日期？
- [ ] 如果文档经历过重大重构，是否保留了旧版备份（带版本标识）？
- [ ] 文档中是否有过时内容？（如描述的系统状态与实际代码不符）

**输出**：列出疑似过时的内容，建议更新或标注。

---

## 输出要求

1. 按上述 6 个检查类别分别输出结果
2. 每类给出：通过项数 / 总检查项数
3. 汇总评分：

| 等级 | 标准 |
|------|------|
| A | 全部通过，结构清晰规范 |
| B | 有小问题（<3 项），不影响理解和执行 |
| C | 有结构性问题（类型混搭/定义重复/关键章节缺失），需重构 |

4. **能直接修复的问题，直接执行修复，不要只列出问题。**
5. 修复完成后重新检查修复项，确认通过。
