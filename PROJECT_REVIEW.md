## A. 项目口述稿（STAR-T 框架）

### 2 分钟精简版

> **背景（S）**：大模型预训练的效果越来越依赖训练数据的质量，但业界对"什么是好的数据清洗方法"缺乏统一量化。不同方法各有论文，但没有在同一数据上做过公平对比。
>
> **任务（T）**：我在同一份 Common Crawl 原始数据上，实现了三代数据清洗方法论——从规则过滤、到模型过滤、再到集成+智能路由+大模型改写的混合方案——并设计了统一的评估体系来做量化对比。
>
> **行动（A）**：整个项目我独立完成了 6000 多行核心代码和 10 个分析 Notebook。工程上最重要的几个设计：一是统一配置系统，所有 Pipeline 读同一份配置文件，一键切换测试模式和正式模式；二是评估分类器和 Pipeline 分类器故意用不同参数，防止"自己评自己"的循环偏差；三是每个阶段都有指标追踪和过滤审计，能定位到具体哪个过滤器误杀了多少文档。
>
> **结果（R）**：第三代方法以 40% 的数据保留率达到了第二代的质量水平，而第二代只有 10% 保留率。也就是说，通过 Pipeline 架构设计的改进，在质量不降的前提下，可用数据量翻了 4 倍。
>
> **收获（T）**：这个项目让我深入理解了数据质量和数据量之间的 trade-off 不是非此即彼的，通过分层路由和 Pipeline 编排可以兼顾。这个思路直接适用于内容安全场景——有害内容的识别和过滤本质上也是一个多阶段、多策略协同的数据 Pipeline 问题。

### 5 分钟详细版

在精简版基础上，展开以下细节：

**技术选型的工程考量**：
- 分类器选了 fastText 而不是 BERT，因为数据清洗场景下速度是瓶颈——fastText 处理 5 万篇文档不到 1 分钟，BERT 要几个小时。在准确率够用（AUC 约 0.85）的前提下，100 倍的速度差异对 Pipeline 吞吐量是决定性的。这和内容安全场景很像：线上审核对延迟极其敏感，模型精度和推理速度的平衡是核心工程问题
- 去重用了两阶段方案：先精确去重（xxhash，零误判），再近似去重（MinHash LSH，可调阈值）。两阶段设计的好处是先用低成本方法筛掉容易的，再用高成本方法处理困难的。这种"由粗到精"的级联思路在大规模数据处理中非常通用

**第三代方法的三个核心机制**：
- **分类器集成**：单个 fastText 分类器偏好百科风格的文本，对技术文档和对话内容打分偏低。我加入了第二个分类器（TF-IDF + 逻辑回归），两者取并集，召回率提升了 15-20%
- **条件性路由（Bypass）**：高质量文档（得分 ≥ 0.7）直接跳过规则过滤——因为我们验证了 18.1% 的高质量文档会被规则误杀（比如技术文档的项目符号比例高，触发了格式规则）
- **大模型改写**：对得分 0.1-0.3 的"可救"文档，用大模型 API 把口语化的内容改写成标准书面语，平均质量分提升 0.15-0.3

**评估体系的设计**：
- 独立于 Pipeline 的评估分类器（不同参数、不同训练数据），防止循环偏差
- 多维度指标：不只看质量分，还看困惑度、毒性、多样性。因为过度过滤会导致数据分布偏移——Gen2 质量最高但多样性最低，基本只剩百科风格的文本
- 过滤审计机制：随机抽样被过滤掉的文档，估算误杀率，帮助定位问题过滤器

## B. 技术深度问答（三层递进）

### 第 1 层：能讲清楚"是什么"

**问：三代数据清洗方法的核心区别是什么？**

> 第一代用人工规则——比如文档长度、字母比例、重复率这些指标——快速过滤明显的垃圾内容。优点是快、可解释，缺点是无法判断语义质量，一篇语法完美但内容空洞的 SEO 文章能通过所有规则。
>
> 第二代训练了一个 fastText 分类器来区分"高质量"和"低质量"文本，只保留得分最高的 10%。质量显著提升，但 90% 的数据被丢弃，数据量可能不够。
>
> 第三代用三个机制解决数据量问题：分类器集成扩大召回、条件性路由减少误杀、大模型改写抢救可用数据。在质量持平的情况下，保留率从 10% 提到 40%。

**问：MinHash LSH 去重的原理是什么？**

> 把文档切成字符级 5-gram，然后用 128 个哈希函数生成一个紧凑签名。签名被分成 8 个 band，每个 band 有 16 行。如果两篇文档在任意一个 band 完全相同，就认为它们是候选重复对，再精确计算 Jaccard 相似度（阈值 0.8）做最终判定。这是一种概率性方法——band 数越多，低相似度的文档对也可能被检出，可以通过调整 band 数和行数来控制精确度和召回率的平衡。

**问：为什么选 fastText 而不是 BERT？**

> 大规模数据清洗场景下，fastText 处理 5 万文档不到 1 分钟，BERT 需要数小时。准确率差距（AUC 0.85 vs 0.92）对数据清洗任务来说可以接受——我们不需要精确到每一篇文档，只需要整体分布足够好。DCLM 论文做过对比实验验证了这一点。

### 第 2 层：能解释"为什么这样设计"

**问：评估分类器为什么必须和 Pipeline 分类器不同？**

> 如果用同一个模型既过滤数据又评估质量，结果必然虚高——相当于考试出题人自己给自己打分。我们的 Pipeline 分类器用 dim=64、bigram（wordNgrams=2），评估分类器故意降到 dim=32、unigram（wordNgrams=1），而且训练数据也不同。这样评估结果才是可信的。这个设计体现了实验设计的基本原则：独立验证。
>
> **映射到内容安全**：如果你用同一个模型同时做内容审核和审核质量评估，也会出现类似偏差。生产环境中需要独立的质量监控体系。

**问：条件性路由的阈值 0.7 和 0.3 是怎么确定的？**

> 来自 Nemotron-CC 论文的实验：他们发现在 0.7 以上的文档中，有 18.1% 会被规则过滤误杀。这些文档格式特殊（比如技术文档大量使用项目符号，触发了"bullet 行比例"规则）但内容优质。0.3 是另一个拐点——低于 0.3 的文档大概率是真正的低质量内容，改写成本高于收益。
>
> 工程实践中这些阈值不是固定的，需要在自己的数据上做阈值调优实验。我在代码中实现了 `compute_bypass_value()` 方法来量化验证"Bypass 挽回了多少文档"。

**问：统一配置系统的设计思路是什么？**

> 所有 Pipeline 和 Notebook 读同一份 `run_config.yaml`，通过 `run_mode` 字段切换"测试模式"（1000 条，10 分钟跑完）和"正式模式"（5 万条，2-3 小时）。好处有三个：
> 1. 消除参数散落——不会出现"这个脚本用 1000 条、那个脚本用 500 条"的不一致
> 2. 降低调试成本——改一个字段就能全局切换
> 3. 保证可复现——配置文件进 Git 版本控制，任何人拉代码都能复现结果
>
> **映射到内容安全**：线上审核系统的灰度发布也需要类似的配置中心——同一套策略可以通过配置切换"全量"和"小流量"。

### 第 3 层：能讨论 trade-off

**问：Gen2 质量最高但保留率只有 10%，这是好事还是坏事？**

> 取决于场景。如果训练数据预算是 5 万亿 token 以内，Gen2 是最好的选择——少而精。但如果需要 15 万亿 token（训练大参数模型），Gen2 就不够了，因为 Chinchilla 定律要求 token 数 ≈ 20 × 模型参数。更关键的隐患是分布偏移：Gen2 过度偏向百科风格，对话、新闻、技术内容被大量过滤，导致模型在这些领域能力不足。
>
> **映射到内容安全**：过度激进的内容过滤也有同样问题——误杀正常内容（false positive）会伤害用户体验。安全策略需要在"漏放有害内容"和"误杀正常内容"之间找平衡。

**问：大模型改写的成本值得吗？**

> 改写一篇文档的 API 调用成本远高于规则过滤或分类器打分。但要算 ROI：改写能把质量分 0.1-0.3 的文档提升到 0.4-0.6，相当于"回收"了本来要丢弃的数据。如果获取新的高质量数据成本更高（比如人工标注），那改写就是划算的。
>
> 工程上的优化：只改写"可救的"区间（0.1-0.3），不浪费钱去改写太差的（< 0.1）或本来就不错的（> 0.3）。这种分层处理的思路在内容安全中也适用——简单 case 用规则快速处理，复杂 case 才送到高成本模型。

**问：集成策略用"并集"还是"交集"？**

> 并集（Union）的召回率更高——38% vs 12%——但引入了更多边界 case。交集（Intersection）精确度更高但太严格，丢弃了太多好的数据。我们选并集是因为后续还有 Heuristic 过滤兜底，多召回一些不会造成灾难性后果。
>
> **映射到内容安全**：多个审核模型的结果合并，也要做类似决策。如果是高风险场景（比如儿童安全），用并集——宁可多拦不可漏；如果是低风险场景（比如轻微敏感词），用交集——减少误伤。

## C. 内容安全 + 数据 Pipeline 场景映射

这一节把 fineweb-pipeline 的工程经验系统性地映射到 TikTok 内容安全业务，展现"可迁移的工程能力"。

### C1. 有害内容过滤 Pipeline

**你的项目经验 → 内容安全映射**：

| fineweb-pipeline 中的做法 | 内容安全场景的对应 |
|---|---|
| 七个过滤器级联执行，每个独立可关闭 | 审核策略模块化：先关键词匹配，再规则引擎，再模型打分，每层可独立调整 |
| 条件性路由（高分跳过规则） | 高置信度内容直接通过/拦截，中间地带送人审 |
| 过滤审计（抽样检查误杀率） | 审核质量监控：定期抽检"被放行的"和"被拦截的"，评估漏放率和误杀率 |
| 阈值调优实验 | 审核阈值调优：不同品类（暴力/色情/仇恨言论）需要不同灵敏度 |
| 统一 stage_tracker 记录每阶段指标 | 审核链路追踪：从内容提交到最终处置，全链路可观测 |

**典型技术问答**：

> **问：如果让你设计一个内容安全审核 Pipeline，你会怎么做？**
>
> 我会设计一个多级级联 Pipeline，核心思路是"由粗到精、由快到慢"：
>
> 1. **第一级：规则引擎**（毫秒级）——关键词黑名单、正则匹配已知有害模式、URL 黑名单。成本最低，处理最明确的 case
> 2. **第二级：轻量模型**（十毫秒级）——类似我项目中的 fastText，对文本做快速分类。高置信的直接处置（通过或拦截），中间地带往下走
> 3. **第三级：重模型**（百毫秒级）——多模态模型（文本+图片+视频帧），处理复杂 case
> 4. **第四级：人工审核**——模型不确定的送人审
>
> 关键工程设计：
> - 每一级都有独立的指标监控（类似我项目中的 stage_tracker），可以看到每级的通过率、拦截率、耗时
> - 阈值是可配置的（类似我项目中的统一配置系统），不同内容品类可以有不同的灵敏度
> - 有过滤审计机制——定期从"放行队列"和"拦截队列"各抽样，人工评估漏放率和误杀率

### C2. 多语言数据处理

**你的项目经验 → 多语言映射**：

| fineweb-pipeline 中的做法 | 内容安全场景的对应 |
|---|---|
| 中文扩展（gen1_zh）：字符级适配（中文没有空格分词） | TikTok 全球化运营，中日韩泰越等语言的文本预处理完全不同 |
| 语言检测（langdetect + fastText 双重） | 内容审核第一步：识别语言，再路由到对应语言的审核策略 |
| 质量规则的语言适配（中文词数统计、停用词替换） | 不同语言的有害内容特征不同（如中文谐音梗、日文假名变体） |

**典型技术问答**：

> **问：多语言内容审核的难点在哪里？**
>
> 我在项目中做中文扩展时遇到过几个典型问题，可以类比到内容安全：
>
> 1. **分词差异**：英文以空格分词，中文需要专门的分词工具（jieba 等），日文更复杂（混合平假名/片假名/汉字）。数据清洗的规则不能直接跨语言复用，需要逐语言适配
> 2. **特征差异**：英文可以用 n-gram 做特征，中文的字符级 n-gram 和词级 n-gram 效果差异很大。我在中文扩展中把质量过滤的统计指标全部从 word-level 改成了 character-level
> 3. **标注数据稀缺**：英文有 Wikipedia 这样的高质量正样本。小语种可能缺乏类似的标注资源，需要考虑跨语言迁移或主动学习策略
> 4. **对抗绕过**：中文谐音字、拼音缩写、火星文；日文半角/全角切换；这些在内容安全场景下是典型的对抗手段，Pipeline 需要做文本标准化（normalization）预处理

### C3. 标注数据管线

**你的项目经验 → 标注管线映射**：

| fineweb-pipeline 中的做法 | 标注管线的对应 |
|---|---|
| 正负样本构造（Wikipedia vs 原始 CC） | 标注数据的来源设计：什么作为正例？什么作为负例？样本比例如何控制？ |
| fastText 快速训练 → 全量数据打分 → 抽样验证 | 模型训练 → 全量推理 → 人工抽检的数据闭环 |
| 大模型改写扩增数据（Synthetic Data） | 标注数据不足时，用大模型生成合成样本做数据增强 |
| 评估分类器独立于 Pipeline 分类器 | 标注质量评估必须独立于标注流程本身 |
| 随机种子固定 + 配置版本控制 | 标注实验可复现，支持回溯和审计 |

**典型技术问答**：

> **问：你怎么保证标注数据的质量？**
>
> 这和我在项目中保证评估公平性的思路是一致的——核心是"独立验证"：
>
> 1. **标注一致性检查**：同一条数据发给多人标注，计算一致率（Kappa 系数）。低于阈值的重新标注
> 2. **黄金标准题**：在标注任务中混入已知答案的标注（类似我项目中的 audit 抽样），监控标注员准确率
> 3. **评估独立性**：评估标注质量的指标不能用标注出来的数据本身去算——这就是我项目中"评估分类器 ≠ Pipeline 分类器"的原则
> 4. **版本控制**：标注数据的每个版本都要快照存档，配合随机种子确保可复现。如果模型效果回退，能追溯到具体哪批标注数据有问题

> **问：标注数据不够怎么办？**
>
> 我在项目中实践过一个方法——大模型合成改写（Synthetic Rephrasing）。具体做法：
>
> 1. 用少量高质量标注数据训练一个初版模型
> 2. 用这个模型对全量数据做预标注
> 3. 人工只审核模型不确定的样本（主动学习），降低标注成本
> 4. 对于确实缺乏标注的品类，用大模型生成合成样本做数据增强
>
> 关键注意点：合成数据有分布偏差风险（类似我项目中 Gen2 偏向百科风格的问题），需要在真实数据上做验证。

## D. 系统设计题

### 设计题 1：设计一个内容安全数据清洗系统

**答题框架**（建议在白板上边画边讲，5-8 分钟）：

```
                    ┌──────────────┐
                    │  内容提交入口  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   语言检测    │  → 路由到对应语言的策略
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼───┐ ┌─────▼─────┐
        │  中文策略  │ │英文策略│ │ 其他语言  │
        └─────┬─────┘ └───┬───┘ └─────┬─────┘
              │            │            │
              └────────────┼────────────┘
                           │
                    ┌──────▼───────┐
            ┌───── │  级联审核引擎  │ ─────┐
            │      └──────────────┘      │
            │                            │
     ┌──────▼──────┐              ┌──────▼──────┐
     │ 规则引擎(L1) │              │  快速模型(L2) │
     │ 关键词/正则  │              │  fastText 等  │
     └──────┬──────┘              └──────┬──────┘
            │                            │
            │      ┌──────────────┐      │
            └───── │ 结果聚合+路由 │ ─────┘
                   └──────┬───────┘
                          │
            ┌─────────────┼─────────────┐
            │             │             │
     ┌──────▼──────┐ ┌───▼────┐ ┌──────▼──────┐
     │  直接通过    │ │ 送人审  │ │  直接拦截   │
     │（高置信安全）│ │（不确定）│ │（高置信违规）│
     └─────────────┘ └────────┘ └─────────────┘
```

**关键设计点**（结合你的项目经验阐述）：

1. **级联架构**（对应项目中七个过滤器级联）
   - 每一级处理特定类型的 case，从简单到复杂
   - 前面的级越快越好，把大部分"明确的"case 处理掉

## E. 高频行为问题

### E1. 项目挑战类

**问：这个项目中你遇到的最大技术挑战是什么？**

> 最大的挑战是设计一个公平的评估体系。听起来简单，但当你用分类器过滤数据、又用分类器评估质量时，很容易陷入循环偏差——用同一个标准既做裁判又做运动员，结果一定好看但没有实际意义。
>
> 我的解决方案是从根本上分离：评估分类器和 Pipeline 分类器使用不同的维度（32 vs 64）、不同的特征粒度（unigram vs bigram）、不同的训练数据。这不是拍脑袋决定的——我做了对比实验，验证了即使参数降低，评估分类器的区分能力依然足够。
>
> 这个经验的迁移意义：在内容安全领域，模型审核质量的评估同样不能用模型本身来做，需要独立的质量监控体系。

**问：你做过的最有影响的工程优化是什么？**

> 统一配置系统。之前每个脚本各自定义数据量、采样数，改一个参数要改五个文件。我设计了一个 `run_config.yaml`，所有 Pipeline 和 Notebook 都读它，`run_mode` 一键切换测试和正式模式。
>
> 这个改动看起来简单，但效果是：调试时间从"每次改参数半小时"降到"改一个字段 10 秒"。更重要的是保证了可复现性——配置进 Git 版本控制，任何人拉代码都能复现完全一样的结果。

### E2. 决策判断类

**问：你在项目中做过什么重要的取舍（trade-off）？**

> 最重要的取舍是分类器集成的策略选择：并集 vs 交集。
>
> 并集召回率更高（38%），但会引入更多边界 case。交集精确度更高，但只有 12% 保留率，太严格了。我选了并集，理由是：后续还有 Heuristic 过滤兜底，边界 case 不会直接污染最终数据。
>
> 这个决策的思路是"看下游有没有兜底机制"。如果这是最后一道防线，我会选交集（宁缺毋滥）。如果后面还有检查，就选并集（先保住召回率）。这个思路在内容安全中也适用——是否宁可误杀，取决于后面有没有申诉和复审机制。

**问：如果项目时间只有一半，你会砍掉什么？**

> 砍掉大模型改写（Gen3 的第三个机制）。原因：
> 1. API 调用成本高，依赖外部服务稳定性
> 2. 只对 0.1-0.3 分的文档有效，覆盖面有限
> 3. 前两个机制（集成 + Bypass）已经能把保留率从 10% 提升到约 30%
>
> 但会保留分类器集成和条件性路由——这两个是纯计算层面的优化，成本低、效果好、工程可靠。

### E3. 协作沟通类

**问：你怎么向非技术的同事解释这个项目？**

> 我会打个比方：想象你要从一大堆混在一起的信件中挑出有价值的。第一代方法是看信封——地址不对的、破损的直接扔掉，但内容好不好看不出来。第二代方法是请一个专家逐封读，只留最好的 10%——质量高但太多信被扔了。第三代方法是请两个专家一起看（互补盲区），对信封没问题的好信直接保留不再过规则（减少误扔），对内容差但主题好的信请人润色一遍再评估。结果：质量一样好，但保留了 4 倍多的信。

**问：你怎么确定项目的优先级？**

> 我用"影响面 × 实施成本"的矩阵来排序。比如在这个项目中：
> - 统一配置系统：影响所有模块、实施成本低 → 最先做
> - 循环偏差防止：影响评估可信度、实施成本中等 → 紧接着做
> - 大模型改写：影响有限（只针对 0.1-0.3 区间）、成本高（API 费用）→ 最后做
>
> 到了内容安全场景：先建可观测性（不知道问题在哪就无法优化），再做规则引擎（覆盖明确 case），最后上模型（处理复杂 case）。
