# 通用方法论模式库

> **适用范围**：数据 pipeline、ML 实验、技术方案设计。从 fineweb-pipeline 项目实践中提炼，可复用到其他项目。
>
> **更新机制**：每次会话结束前，检查是否有新的方法论洞察需要补充（见文末触发清单）。

---

## 1. 指标口径先行

**问题**：方案中写"过滤率 20%"、"质量提升 10%"，但没说分子分母是什么，导致不同环节的数字无法对比，甚至自相矛盾。

**规则**：

- 任何百分比/比率出现时，**必须同时定义分子和分母**
- 条件概率 vs 联合概率是最常见的混淆源：
  - 条件过滤率 = 该步丢弃数 / **该步输入数**（"到达这一步的文档中多少被过滤"）
  - 绝对过滤量 = 该步丢弃数 / **原始总数**（"这一步对整体贡献多大"）
- 绝对差 vs 相对变化率也容易混："+10%" 是从 0.3 涨到 0.4（绝对差 0.10），还是涨了 10%（从 0.3 到 0.33）？

**实践模板**：

```markdown
| 指标名 | 公式 | 分子 | 分母 | 示例 |
|--------|------|------|------|------|
| XXX率  | A/B  | A=...| B=...| 当A=100,B=500时，XXX率=20% |
```

**检查清单**：
- [ ] 方案中每个百分比都能回答"分子是什么？分母是什么？"
- [ ] 同一指标在不同章节的口径是否一致
- [ ] 绝对差和相对变化率是否有明确标注

## 2. 数据采样抗偏斜

**问题**：从大数据集中采样时，顺序读取单一分片会导致域名集中、语言偏斜、质量同质化。

**规则**：

- **多源分层 > 单源顺序**：从 N 个分片各取 K/N 条，而非从 1 个分片取 K 条
- **全局 shuffle**：合并后打乱顺序，打破源内的局部相关性
- **验证多样性**：采样后统计来源分布、域名数、语言比例，确认无严重偏斜

**决策框架**：

```
数据量需求 < 单分片容量？
  ├─ 是 → 仍然建议多源采样（至少 5-10 个分片）
  └─ 否 → 必须多分片，且需容错（单分片失败自动补选）

采样后验证：
  ├─ 独立来源数 ≥ 目标的 80%
  ├─ 最大单一域名占比 < 5%
  └─ 语言/类型分布与全量统计无显著偏差
```

**适用场景**：
- 从 Common Crawl 采样训练/评估数据
- 从日志系统采样做分析
- 从多个数据库表采样做联合分析

## 3. 分模块拆解复合指标

**问题**：端到端指标（如"总保留率 15%"）无法定位瓶颈，也无法与论文对标。

**规则**：

- **串行 pipeline**：总保留率 = 各步存活率的乘积。必须拆解到每一步，才能发现哪步损耗最大
- **并行模块**：总输出 = 各模块输出之和。必须拆解到每个模块的贡献占比
- 每个模块需要三列：**输入量、该模块处理逻辑、输出量**

**拆解模板（串行）**：

```
输入（100%）
  → 步骤 A（条件过滤率 X%）    剩余 Y%
  → 步骤 B（条件过滤率 X%）    剩余 Y%
  → ...
  ─────────────────────────
  输出 ≈ Z%

| 步骤 | 输入(% of 原始) | 条件处理率 | 绝对贡献量 | 输出(% of 原始) |
```

**拆解模板（并行/分桶）**：

```
输入（100%）
  → 统一打分/分类
      ├─ 桶 A（X%）→ 处理方式 1 → 通过率 P1
      ├─ 桶 B（Y%）→ 处理方式 2 → 通过率 P2
      └─ 桶 C（Z%）→ 处理方式 3 → 通过率 P3

| 模块 | 输入占比 | 通过率 | 输出占比 | 说明 |
```

**价值**：
- 定位瓶颈（哪一步损耗最大）
- 对标论文（论文的 X% 对应我们的哪一步）
- 调参指导（想提高整体保留率，应该调哪一步）

**拆解后必须纳入结果校验**：

拆解不能只停留在方案设计阶段，必须在执行验证的核对清单中逐模块检查。具体要求：

1. **每步都要有绝对数字**：不能只验证"整体保留率 30-50%"，还要验证每步输出的具体文档数。百分比可能掩盖绝对量级问题。
2. **内部一致性交叉校验**：
   - 串行 pipeline：(1−过滤率₁) × (1−过滤率₂) × ... ≈ 整体存活率？偏差应 <5%
   - 并行/分桶：各桶文档数之和 = 输入总数？各模块输出之和 = 总输出？
3. **每个模块独立设预期值和红线**：不能只在整体层面设红线。某步过滤率异常（如语言过滤率 <1%）可能被整体存活率"平均掉"，但实际说明该步过滤器未生效。

**核对清单模板（串行 pipeline）**：

```markdown
各步条件过滤率：
| 步骤 | 口径 | 预期值 | 实际值 | 偏差 | 建议 |

各步输出文档数（具体场景 N=X）：
| 步骤 | 口径 | 预期值 | 实际值 | 偏差 | 建议 |

整体指标 + 内部一致性：
| 整体存活率 | ... |
| 内部一致性 | 各步存活率乘积 ≈ 整体存活率？ | 偏差<5% | ... |
```

**核对清单模板（并行/分桶）**：

```markdown
各桶分布：
| 桶名 | 口径 | 预期占比/文档数 | 实际值 | 偏差 | 建议 |
| 各桶之和 = 输入总数？ | ... |

各模块输出：
| 模块 | 输出文档数 | ... |
| 总输出 = 各模块输出之和？ | ... |
```

## 4. 论文基准换算

**问题**：直接引用论文数字作为自己的预期值，但忽略了输入数据层级、处理步骤、规模差异。

**规则**：

- **识别论文 pipeline 的完整步骤链**，明确每步的输入输出
- **标注自己跳过了哪些步骤**，将跳过步骤的存活率设为 100%
- **倒推可比指标**：自己的预期值 = 论文端到端值 / (跳过步骤的存活率乘积)

**换算模板**：

```
论文 pipeline：A → B → C → D → 输出（端到端保留率 X%）
我们的 pipeline：    B → C → D → 输出（跳过了 A）

论文各步存活率估计：A=a%, B=b%, C=c%, D=d%
  验证：a × b × c × d ≈ X%

我们的预期值：b × c × d = X% / a%

| 差异因素 | 论文 | 我们 | 对保留率的影响 |
```

**注意事项**：
- 论文数字通常是特定规模/配置下的结果，小规模复现可能偏差较大
- 明确标注哪些差异会导致偏高、哪些偏低，给出预期范围而非点估计

## 5. 样本量倒推法

**问题**：凭直觉设定采样量（如"先下 1000 条试试"），导致下游环节样本不足、统计无意义。

**规则**：

- **从下游最苛刻的需求倒推上游输入量**
- 考虑逐级衰减：如果 pipeline 每步存活率 40%、10%，那么下游需要 100 条，上游需要 100 / 40% / 10% = 2500 条
- 设计多档体系：快速验证档（验证代码不报错）、统计有效档（结果有意义）、展示档（最终结果）

**倒推模板**：

```
| 下游环节 | 最小样本需求 | 逆推路径 | 对应上游输入量 |
|---------|------------|---------|--------------|
| 环节 X  | N 条       | N ÷ 存活率1 ÷ 存活率2 | M 条 |

瓶颈环节 = 需要上游输入量最大的那个
各档位设计依据 = 能覆盖哪些环节的最小需求
```

## 6. 数据源必须匹配任务目标

**问题**：优先考虑"好不好下载"而非"适不适合当输入"，导致整个实验缺乏意义。

**规则**：

- 选数据源前先问：**"我的 pipeline 期望输入什么质量层级的数据？"**
- 清洗 pipeline 的输入应该是脏数据（否则清洗无效果可展示）
- 评估 pipeline 的输入应该有已知标签（否则无法验证正确性）

**决策框架**：

```
任务类型          → 需要的数据特征        → 错误选择的后果
数据清洗 pipeline → 脏数据（未经过滤）     → 在干净数据上跑清洗，指标全部趋近 0
质量分类器训练    → 有区分度的正负样本     → 正负样本太相似，AUC 无法提升
去重效果验证      → 包含一定重复的数据集   → 无重复数据上跑去重，去重率永远 0%
```

## 7. 执行验证闭环

**问题**：生成代码/方案后标记"完成"，但实际从未执行验证。

**规则**：

- **"完成"的定义 = 代码已执行 + 输出已验证 + 错误已修复**
- 每个阶段设置核对清单：预期值 vs 实际值 → 偏差分析 → 下一步建议
- 遇到红线指标必须停下修复，不能跳过继续

**核对清单模板**：

```markdown
| 检查项 | 口径 | 预期值 | 实际值 | 偏差方向 | 建议 |
|--------|------|--------|--------|---------|------|
| 指标 A | 分子/分母 | X | ? | 偏高/偏低/正常 | 继续/调参/排查 |
```

## 8. 核对清单驱动的偏差反馈循环

**问题**：即使有核对清单，如果只是"填完就过"，偏差不会被修复，后续阶段会在错误基础上继续。

**核心机制**：每个 Phase 结束后，不是简单打勾，而是执行完整的闭环：

```
预设预期值 → 执行 → 填入实际值 → 对比偏差 → 评估方向 → 决策
                                                          │
                    ┌─────────────────────────────────────┘
                    │
              ├─ 在预期范围内 → 继续下一 Phase
              ├─ 偏离但量级合理 → 记录原因，在分析中标注，继续
              └─ 触发红线 → 停下修复，重跑当前 Phase，不跳过
```

**设计要点**：

1. **预期值必须在执行前设定**，不能事后"倒推合理化"。预期值来自论文基准换算（见模式 4）或样本量倒推（见模式 5）
2. **红线指标是硬性门禁**，触发后必须修复才能进入下一阶段。非红线指标允许带偏差继续，但必须记录
3. **偏差方向比绝对值更重要**：偏高和偏低的原因和对策完全不同，核对清单中需要分方向给出建议
4. **阶段间有数据依赖**：前一阶段的实际输出量决定后一阶段的可行性。例如 Gen1 输出条数决定了 Gen2 训练集是否足够

**预期值 vs 红线的区别**：

| | 预期值 | 红线 |
|---|--------|------|
| 性质 | 最可能的结果范围 | 不可接受的极端情况 |
| 触发后果 | 偏离时记录分析原因 | 触发时必须停下修复 |
| 设定依据 | 论文基准换算 + 经验估计 | 逻辑上不可能或代码 bug 的信号 |
| 示例 | Gen1 存活率 30-50% | Gen1 存活率 <10%（阈值过严）或 >80%（数据不够脏） |

**多阶段联动**：

后一阶段的预期值应基于前一阶段的**实际值**而非预期值来调整。例如：
- 预期 Gen1 存活率 40%，实际 35% → Gen2 预期输入量从 4800 调整为 4200
- 如果调整后不满足 Gen2 最小样本需求 → 需要追加下载数据

**适用场景**：
- 多阶段 ML pipeline（训练→评估→部署）
- 数据工程（ETL 各步验证）
- 任何有质量门禁的流水线

## 9. 英文缩写首次全写

**问题**：文档中直接使用 AUC、ROC、MinHash、ETL 等缩写，非该领域读者无法理解；即使是专业读者，同一缩写在不同上下文可能含义不同（如 CC = Common Crawl 还是 Creative Commons）。

**规则**：

- **首次出现时必须全写 + 中文简释**，后续可只用缩写
- 格式：`全写（缩写，简释）`，例如 `Area Under the ROC Curve（AUC，分类器区分正负样本的能力，1.0=完美，0.5=随机猜测）`
- 如果文档较长（>3 页），在文档开头或附录设置**术语表**

**模板**：

```markdown
| 缩写 | 全写 | 简释 |
|------|------|------|
| AUC | Area Under the ROC Curve | 分类器区分正负样本的能力，1.0=完美，0.5=随机猜 |
| CC | Common Crawl | 非营利组织，定期爬取互联网并公开数据 |
| WET | WARC Encapsulated Text | CC 提供的纯文本格式，已去除 HTML 标签 |
```

**检查清单**：
- [ ] 搜索文档中所有大写连续字母（`[A-Z]{2,}`），确认每个缩写至少有一次全写
- [ ] 同一缩写在文档中是否始终指同一含义
- [ ] 跨领域读者能否仅凭文档理解所有缩写（不需要外部搜索）

## 10. 文档逻辑性检查

**问题**：技术文档写长后，章节间逻辑链断裂、内容重复、结论与推导脱节，读者无法跟上推理过程。

**规则**：

**每章开头声明逻辑链**：用一句话说明"本章从哪来，到哪去"，让读者知道自己在推理链的什么位置。

```markdown
> 本章逻辑：论文数据建立基准线 → 推导我们在 CC WET 上的预期值 → 汇总为可量化验收指标
```

**章节组织的三个检查**：

1. **逻辑链完整性**：每章的结论是否能被前序章节支撑？后续章节是否依赖本章的输出？
   ```
   章节依赖图（示例）：
   背景（定义层级）→ 论文基准（建立锚点）→ 预期值推导（换算到我们的场景）
     → 样本量倒推（从预期值反推输入需求）→ 执行计划（按样本量设计流程）
   ```

2. **内容去重**：同一概念/数据只应有一个权威定义位置，其他位置引用即可。
   - 反例：CC WET 的介绍在"背景"和"数据层级"中各写一遍
   - 正例：在"数据层级"中完整定义，"背景"中引用 `（详见 1.2 节数据层级定义）`

3. **结论可追溯**：每个结论/预期值旁边应标注推导依据。
   - 反例："Gen1 预期保留率 30-50%"（从哪来的？）
   - 正例："Gen1 预期保留率 30-50%（= 论文 10-15% ÷ 提取存活率 30-50% ÷ 去重存活率 80-90%，见 2.1 节推导）"

**文档自检清单**：

- [ ] 每章开头有逻辑链声明吗？
- [ ] 同一概念是否只在一个地方完整定义？
- [ ] 所有数字/结论是否标注了来源（论文引用 or 推导过程）？
- [ ] 删除任何一章后，后续章节是否会引用到不存在的内容？（如果是，说明依赖关系需要标注）
- [ ] 有没有两个章节在说"差不多的事情"？（合并或拆分为引用关系）

## 11. 需求-实现同步（方案文档一致性）

**问题**：方案的"设计章节"新增了需求（如新过滤规则、新处理步骤），但"执行计划章节"没有对应的代码实现步骤。结果方案审批后执行时发现漏做了功能，返工重写。

**规则**：

1. **每次在设计章节新增功能/步骤时，必须同步在执行计划中添加对应的代码实现步骤**
   - 设计章节定义"做什么"（What）
   - 执行计划定义"怎么做"（How）：文件路径、函数名、验收标准
   - 两者必须 1:1 对应

2. **自检清单**（每次修改方案文档后执行）：
   - [ ] 设计章节中的每个处理步骤，在执行计划中是否有对应的代码实现步骤？
   - [ ] 执行计划中的每个步骤，是否有明确的验收标准？
   - [ ] 新增的配置参数（阈值、开关）是否在 config 文件步骤中列出？
   - [ ] 核对清单（C-2 等）是否覆盖了新增步骤的输出验证？

3. **反模式（禁止）**：
   - ❌ 设计章节写了"九类质量规则"，执行计划仍只有"质量过滤"一笔带过
   - ❌ 设计章节写了 PII 脱敏，但没有 Phase A 步骤创建 `pii_removal.py`
   - ❌ 设计章节新增了 URL 去重，但 C-2 核对清单没有 URL 去重率的验证项

**实践模板**：

```
设计章节新增内容 → 对应执行计划更新
───────────────────────────────────────────
新增处理步骤 X  → Phase A: 创建 src/xxx.py，验收 = 单测通过
                → Phase A: 集成到 pipeline.py，验收 = import 不报错
                → Phase A: 新增 config 参数，验收 = YAML 可加载
                → Phase C-2: 核对清单新增 X 的输出验证行
                → 2.3 验收表: 新增 X 的指标行
```

## 12. 阈值必须对标数据分布

**问题**：从论文或代码默认值复制过滤阈值，但未验证这些阈值是否适合当前数据的分布特征，导致过滤器要么杀掉 90%+ 的数据（过严），要么形同虚设（过松）。

**规则**：

- **先测分布，再定阈值**：对目标数据计算指标的 P10/P25/P50/P75/P90 分位数，然后基于分布设定合理阈值
- **阈值与论文数据来源匹配**：论文的阈值是在论文的数据上校准的。数据来源不同（如 CC WET vs FineWeb），阈值必须重新校准
- **观察条件过滤率反推**：如果某步过滤率 >80%，大概率是阈值不匹配数据，而非数据真的那么差

**校准流程**：

```
1. 取采样数据（≥500 条）
2. 计算目标指标的分位数分布
3. 选择阈值使得条件过滤率在合理范围（通常 10-50%）
4. 对比论文原始阈值，记录偏差原因

| 指标 | 论文阈值 | 数据 P50 | 调整后阈值 | 偏差原因 |
```

**实际案例（2026-03-06，fineweb-pipeline）**：

## 13. 配置值传参 vs 硬编码默认值

**问题**：YAML 配置文件中定义了阈值，但代码构造函数使用硬编码默认值，config 值从未被传入。表面看"配置生效了"，实际上改 config 完全无效。

**规则**：

- **构造链追踪**：从 config YAML → config loader → pipeline 初始化 → 子组件构造函数，每一层都要确认参数被传递
- **config 值优先**：如果 config 中定义了值，代码必须读取并传入，不能只用构造函数默认值
- **验证方法**：修改 config 中某个阈值为极端值（如 0.0 或 1.0），运行 pipeline，观察该步过滤率是否变化

**检查清单**：

- [ ] config YAML 中的每个参数，是否都有代码读取它？
- [ ] 读取后是否传入了对应的构造函数？（而非丢弃在中间变量）
- [ ] 构造函数的默认值是否与 config 的默认值一致？（如不一致，说明有 bug）

**反模式（实际案例）**：

```python
# BAD: 读了 config 但只用了 enabled 字段，其他参数全部丢弃
goph_cfg = filter_cfg.get("gopher_quality", {})
self.quality_filter = QualityFilter(
    use_gopher=goph_cfg.get("enabled", True),  # 只传了开关
    # min_alpha_ratio=0.5 在 config 中，但从未传入
)

# GOOD: 将 config 参数透传到子组件
gopher_kwargs = {k: v for k, v in goph_cfg.items() if k != "enabled"}
self.quality_filter = QualityFilter(
    use_gopher=goph_cfg.get("enabled", True),
    gopher_kwargs=gopher_kwargs,
)
```
