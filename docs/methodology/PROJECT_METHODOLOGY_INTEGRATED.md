# 项目执行方法论（整合版）

> 将项目一复盘的真实教训，与业界成熟方法论（MECE、金字塔原理、CRISP-ML(Q)、ML 报告标准）整合。用于指导项目二三的 Claude Code Prompt 设计和 Notebook 规范。

---

## 一、框架先行：先想清楚再动手

### 1.1 五要素定义（每个 Notebook 开头必写）

```
每个 Notebook / 每个 Pipeline 阶段必须先明确：
  ① 输入：什么数据，什么格式，从哪来，多少条
  ② 输出：产出什么，给谁用，什么格式，交付标准
  ③ 中间过程：经过哪些步骤，每步做什么
  ④ 目标：这个环节要达成什么效果
  ⑤ 度量标准：用什么指标衡量，基准值（论文值）是多少，预期效果是多少
```

### 1.2 假设合理性检查

```
每个论证环节的假设必须合理：
  如果假设和论文/业界标准有偏离 →
    ① 明确承认偏离
    ② 给出偏离的合理性解释（如"数据规模不同""硬件限制"等）
    ③ 评估偏离对最终结论的影响程度
  → 不能默默跳过
```

### 1.3 项目一教训

```
  框架初期构思不清晰 → 后续反复修改 → 浪费大量时间
  核心教训：先确保方法论是通的 → 框架打清楚 → 再跑数据
  像做复杂数据分析 → 事先有框架 → 有了框架再输出
```

### 【方法论整合】MECE 原则（McKinsey）

```
MECE = Mutually Exclusive, Collectively Exhaustive（相互独立，完全穷尽）

应用于项目框架设计：

  ① 相互独立（ME）：每个 Notebook 的职责不重叠
     Notebook 03 做文本清洗 / Notebook 04 做多模态清洗
     → 不能两个 Notebook 都做"文本去重"

  ② 完全穷尽（CE）：所有 Notebook 加起来覆盖完整 Pipeline
     输入→清洗→增强→训练→评估→消融
     → 不能遗漏任何关键环节

检查方法：
  画出完整数据流 → 每个环节标注对应的 Notebook
  → 有没有环节没人负责？（不穷尽）
  → 有没有两个 Notebook 做同一件事？（不独立）
```

### 【方法论整合】金字塔原理（Pyramid Principle, Barbara Minto）

```
核心：先说结论，再给论据

应用于 Notebook 结构：

  每个 Notebook 开头第一个 Markdown Cell：
    结论："本 Notebook 证明了 XXX 对安全分类的贡献是 YYY"
    → 读者 3 秒钟知道这个 Notebook 的价值

  然后展开支撑（不超过 3 个主要论点）：
    论点 1：数据准备方法 + 结果
    论点 2：实验设计 + 结果
    论点 3：对比分析 + 结论

  每个论点再展开具体数据和图表

  → 面试时也用这个结构讲项目：先说结论再展开
```

## 二、规范性与方法论的建立

### 2.1 方法论 Notebook（Notebook 00，全局地图）

```
第一个 Notebook 专门放置：
  ① 项目整体架构图（数据流、模型流、评估流）
  ② 所有关键概念定义
  ③ 术语缩写表（英文缩写首次出现标注全称）
  ④ 每个后续 Notebook 的五要素摘要（输入/输出/目标/指标/基准值）
  ⑤ 参考论文和标准值汇总表
```

### 2.2 论文标准值的口径核验

```
如果引用论文方法或结论：
  ① 记录论文原始条件（数据集、模型规模、硬件、训练 epoch 等）
  ② 记录我们的实验条件
  ③ 明确列出差异项
  ④ 评估差异对结果的影响
  ⑤ 给出我们的结果应该落在什么范围内的预期

示例：
  | 对比项 | 论文（DCLM） | 我们的实验 | 影响评估 |
  |--------|-------------|-----------|---------|
  | 数据量 | 240T token  | 50K 文档   | 结论趋势一致但绝对值会低 |
  | 模型   | 7B          | GPT-2 125M | 可能有能力涌现差异 |
  | 训练时长| 完整训练     | 5 epoch    | 效果未充分收敛 |
```

### 2.3 统一指标与术语定义

```
  ① 新指标出现 → 提前在方法论 Notebook 中定义
  ② 英文缩写 → 全文第一次出现标注全称
  ③ 术语表放在 Notebook 00 → 全项目引用同一定义
  ④ 同一指标在不同 Notebook 中计算口径必须一致
```

### 2.4 提前谋划文档结构

```
  先出全局 Pipeline 图 → 确认每步的输入输出
  → 再拆解到每个 Notebook
  → 每个 Notebook 先写 Markdown 框架 → 再写代码
  → 先有全局地图再走路
```

### 【方法论整合】CRISP-ML(Q) 六阶段框架

```
CRISP-ML(Q) = Cross-Industry Standard Process for ML with Quality Assurance
（跨行业 ML 标准流程 + 质量保障方法论）

六个阶段（每个阶段都要有质量检查点）：

  Phase 1：业务理解 + 数据理解（合并）
    → 明确业务目标、成功指标、数据可行性
    → 输出：项目范围定义 + 数据质量报告

  Phase 2：数据准备
    → 清洗、特征工程、数据增强、标准化
    → 输出：训练就绪的数据集 + 数据处理 Pipeline（可复现）

  Phase 3：建模
    → 模型选择、训练、超参调优
    → 输出：候选模型 + 实验记录

  Phase 4：评估
    → 性能评估、鲁棒性测试、可解释性
    → 输出：评估报告 + 对标业务目标的结论

  Phase 5：部署
    → 模型集成到系统、API 封装
    → 对于面试项目：产出可运行的 Pipeline 脚本

  Phase 6：监控与维护
    → PSI/KS 漂移检测、持续更新
    → 对于面试项目：设计监控方案即可（不需要真正部署）

每个阶段的质量保障：
  ① 明确该阶段的风险（数据质量差？过拟合？不可复现？）
  ② 选择对应的 QA 方法（交叉验证？消融实验？可复现性检查？）
  ③ 记录决策和理由
```

### 【方法论整合】ML 报告标准结构

```
基于 CS 391L（UT Austin）ML 报告标准：

  1. Introduction（引言）
     动机 + 问题定义 + 基本方法 + 主要结论预览

  2. Problem Definition & Algorithm（问题定义与算法）
     2.1 精确定义输入输出
     2.2 算法描述 + 伪代码 + 具体示例

  3. Experimental Evaluation（实验评估）
     3.1 方法论：假设是什么？自变量因变量是什么？数据怎么来的？
     3.2 结果：数据和图表，和竞争方法的对比
     3.3 讨论：结果说明了什么？为什么？意外发现？

  4. Related Work（相关工作）
     本方法和已有方法的异同

  5. Conclusion（结论）
     主要发现 + 局限性 + 未来方向

应用到 Notebook：
  每个 Notebook 的 Markdown Cell 按这个结构组织
  → 面试时直接打开 Notebook 讲 = 一份完整的报告
```

## 三、Agent（Claude Code）执行中的核验

### 3.1 命令执行不完整

```
  Agent 可能没考虑到位就反馈完成了
  → 每一步执行后，人工检查实际输出是否符合预期
  → 在 Prompt 中明确要求"执行完后输出验证结果"
```

### 3.2 记忆缺失

```
  长对话中早期指令容易丢失
  → 关键约束写进系统 Prompt / 配置文件 / CLAUDE.md
  → 关键节点重新提醒
  → 使用 /compact 而不是 /clear（保留压缩上下文）
```

### 3.3 逻辑连贯性

```
  跨 Notebook 的一致性容易断裂
  → 建立"检查清单"：每个 Notebook 完成后对照检查
  → 指标定义、数据格式、术语使用是否和 Notebook 00 一致
```

### 【方法论整合】Claude Code Prompt 设计原则

```
基于项目一的经验，给 Claude Code 的 Prompt 应包含：

  ① 全局上下文文件（CLAUDE.md 或方法论 Notebook）
     → Agent 每次启动都能读到全局约束

  ② 每个任务的验收标准
     → "执行完后，检查输出是否满足以下条件：..."

  ③ 论文基准值
     → "这个实验的论文基准是 AUC=0.85，我们的目标是 >0.75"

  ④ 已知陷阱提醒
     → "注意：不要用 accuracy 评估不均衡数据"
     → "注意：CLIP 是特征提取器不做匹配"

  ⑤ 输出格式要求
     → "每个实验结果用表格记录：实验名/设置/指标1/指标2/结论"
```

## 四、串行任务的标准化

### 4.1 每个节点的标准定义

```
Pipeline 中每一个节点必须有：
  ① 输入标准：期望的数据格式、数据量、质量要求
  ② 处理逻辑：这一步具体做什么（简要描述）
  ③ 输出标准：产出的数据格式、数据量、质量要求
  ④ 基准值：论文/业界这一步的标准结果是什么
  ⑤ 预期值：我们期望达到多少（可以低于论文，但要说明为什么）
  ⑥ 验收检查：怎么确认这一步做对了
  ⑦ 观察重点：这一步最可能出问题的地方
```

### 4.2 Pipeline 节点卡片模板

```
| 项目 | 内容 |
|------|------|
| 节点名称 | 例：DistilBERT 文本安全分类器训练 |
| 输入 | safety_sft_cleaned.jsonl, 14 类标签, ~80K 条 |
| 输出 | distilbert_safety_model/ + eval_report.json |
| 处理逻辑 | DistilBERT + 分类头, Focal Loss, 3 epochs |
| 基准值 | WildGuard 论文 AUC=0.89 |
| 预期值 | AUC > 0.80（数据量较少，允许低 10%） |
| 偏离说明 | 论文用 WildGuard 专用模型 7B，我们用 DistilBERT 66M |
| 验收检查 | 验证集 AUC > 0.80 且 14 类 Recall 均 > 60% |
| 观察重点 | 稀有类别（儿童安全、政府决策）的 Recall |
| 失败处理 | 如 AUC < 0.75 → 检查数据配比 → 加权 Loss 调参 |
```

### 【方法论整合】SCR 叙事结构（McKinsey）

```
SCR = Situation → Complication → Resolution（情境→矛盾→解决）

应用于每个 Notebook 的"为什么做这个实验"：

  Situation（背景）：
    "安全分类器需要处理 14 个类别，但数据极度不均衡"

  Complication（矛盾）：
    "普通 Loss 训练后，稀有类别 F1 只有 0.3"

  Resolution（解决）：
    "使用 Focal Loss + 难例挖掘，稀有类别 F1 提升到 0.65"

  → 每个实验都有清晰的"为什么做→发现什么问题→怎么解决"
  → 面试时讲项目也用这个结构
```

## 五、实验记录标准

### 5.1 每个实验必须记录

```
  ① 实验目的：验证什么假设
  ② 实验设置：改了什么变量，其他变量怎么控制
  ③ 基准对比：和什么比（论文值 / 上一版 / baseline）
  ④ 结果数字：关键指标的具体数值
  ⑤ 结论：一句话说明结果支持还是否定假设
  ⑥ 意外发现：有没有预期之外的观察（往往最有价值）
```

### 5.2 实验记录表模板

```
| 实验名 | 设置 | AUC | Recall | Over-refusal | 结论 |
|--------|------|-----|--------|-------------|------|
| Baseline | 普通 CE Loss | 0.82 | 91% | 5% | 基准 |
| +加权 Loss | 权重=类别倒数 | 0.84 | 93% | 7% | Recall 提升但 Over-refusal 也升 |
| +Focal Loss | γ=2 | 0.85 | 94% | 5% | 最优：Recall 提升且 Over-refusal 不变 |
| -对比数据 | 去掉 WildJailbreak | 0.80 | 95% | 25% | 验证对比构造必要性 |
```

### 【方法论整合】可复现性清单（CRISP-ML(Q)）

```
每个实验必须记录以下元信息（确保可复现）：

  ☐ 代码版本（Git commit hash）
  ☐ 数据版本（数据文件的 hash 或版本号）
  ☐ 随机种子（seed=42 或其他固定值）
  ☐ 超参数完整列表（lr, batch_size, epochs, β 等）
  ☐ 运行环境（Python 版本、GPU 型号、关键库版本）
  ☐ 训练时长
  ☐ 结果是否在不同 seed 下稳定（至少跑 2 次）
```

## 六、对项目二三的指导原则（总结）

```
  ① 先出方法论 Notebook（Notebook 00）→ 确认后再执行代码
  ② 每个 Notebook 先写五要素 Markdown → 再写代码
  ③ 用 MECE 检查框架完整性（不重叠、不遗漏）
  ④ 用金字塔原理组织每个 Notebook 的叙事（结论先行）
  ⑤ 用 SCR 叙事每个实验（背景→矛盾→解决）
  ⑥ 论文标准值必须引用并做口径核验
  ⑦ 术语和指标统一定义，全文一致
  ⑧ Claude Code Prompt 包含全局约束 + 验收标准 + 基准值
  ⑨ 串行 Pipeline 每个节点有标准卡片
  ⑩ 实验记录必须包含目的/设置/数字/结论 + 可复现性元信息
```
