# 项目一复盘：核心经验与方法论要点

> 基于项目一（Pre-train 数据清洗）全流程执行中的真实教训，提炼出的核心原则。用于指导项目二、项目三的执行，避免重复犯错。

---

## 一、框架先行：先想清楚再动手

### 1.1 整体框架必须提前确定

```
每个项目/每个 Notebook 必须先明确五要素：
  ① 输入：什么数据，什么格式，从哪来
  ② 输出：产出什么，给谁用，什么格式
  ③ 中间过程：经过哪些步骤，每步做什么
  ④ 目标：这个环节要达成什么效果
  ⑤ 度量标准：用什么指标衡量，基准值是多少，预期效果是多少
```

### 1.2 每个论证环节的假设必须合理

```
  如果某个假设和论文/业界标准有偏离：
    → 必须明确承认偏离
    → 给出偏离的合理性解释
    → 不能默默跳过
```

### 1.3 项目一暴露的问题

```
  (a) 框架初期构思不清晰 → 后续反反复复修改
  (b) 很多问题前前后后不断调整 → 浪费大量时间
  (c) 一开始没想清楚某个方法要不要用、合理性是什么
      → 事后再调整对整体影响很大

  核心教训：
    必须先确保方法论是通的 → 把框架打清楚 → 再去跑数据
    这就像做复杂的数据分析 → 事先一定要有框架 → 有了框架再输出内容
```

## 二、规范性与方法论的建立

### 2.1 方法论 Notebook（第一个 Notebook）

```
  专门放置详细的方法论介绍：
    覆盖所有关键概念定义
    覆盖整体框架和数据流
    覆盖每个环节的输入/输出/预期/度量标准
```

### 2.2 论文标准值的口径核验

```
  如果用到论文里的方法或结论：
    → 必须把论文的标准值放进去
    → 做好口径核验（我的实验条件和论文是否一致）
    → 如果不一致，明确说明差异和影响
```

### 2.3 统一指标与术语定义

```
  (a) 新指标出现时 → 提前说明并做好定义
  (b) 英文缩写 → 第一次出现时标注全称
  (c) 大写术语 → 提前做好缩写定义表
  → 全文保持术语一致性
```

### 2.4 提前谋划整个文档结构

```
  从大处着眼：
    第一步：搞定方法论
    第二步：明确每个环节的输入/输出/预期/观察重点
    第三步：先出一个大的 Pipeline → 确认后再执行
  → 先有全局地图再走路，效果远好于边走边看
```

## 三、Agent（Claude Code）执行中的核验问题

### 3.1 命令执行不完整

```
  下达命令后 → Agent 可能没考虑到位就直接反馈完成了
  缺乏进一步的核验
  → 需要人工检查每一步的实际输出是否符合预期
```

### 3.2 记忆缺失

```
  前面提到的问题/约束 → Agent 后面可能就忘掉了
  尤其在长对话中，早期的指令和约束容易丢失
  → 需要在关键节点重新提醒
  → 或者把约束写进配置文件/系统 Prompt 中持久化
```

### 3.3 逻辑连贯性

```
  文档中间的逻辑性和连贯性容易断裂
  Agent 可能在某个 Notebook 中做了正确的事
  但在下一个 Notebook 中没有保持一致
  → 需要跨 Notebook 的一致性检查
```

## 四、串行任务的标准化

### 4.1 每个节点必须有标准

```
  串行 Pipeline 中的每一个节点：
    ① 输入标准：期望的数据格式和质量
    ② 处理逻辑：这一步具体做什么
    ③ 输出标准：产出的数据格式和质量
    ④ 指标标准值：论文/业界的基准是多少
    ⑤ 预期值：我们期望达到多少
    ⑥ 观察重点：这一步最可能出问题的地方是什么
```

### 4.2 形成规范流程

```
  不是"做完一步看看再决定下一步"
  而是"整个 Pipeline 的每一步都提前规划好"
  每一步的验收标准都提前定义
  → 执行时只需要对照标准检查
  → 而不是临时决定"这个结果算不算好"
```

---

## 五、对项目二三的指导原则（总结）

```
  ① 先出完整的方法论 Notebook → 确认后再执行
  ② 每个 Notebook 先写五要素（输入/输出/过程/目标/度量）→ 再写代码
  ③ 论文标准值必须引用并核验口径
  ④ 术语和指标统一定义，全文一致
  ⑤ Agent 执行后必须人工核验关键输出
  ⑥ 串行 Pipeline 每个节点有明确的验收标准
  ⑦ 先有全局地图再走路
```
