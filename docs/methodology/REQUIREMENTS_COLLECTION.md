## 阶段一：项目初始化（2026-03-05）

### R01. 代码生成 ≠ 任务完成

| 维度 | 内容 |
|------|------|
| **场景** | 阶段一~四标记"完成"，创建了 run_gen1.py、run_gen2.py、run_gen3.py 等脚本 |

## 阶段六：数据源替换（2026-03-06）

### R03. 数据源必须匹配任务目标

| 维度 | 内容 |
|------|------|
| **场景** | 用 FineWeb（已清洗，垃圾率~5%）作为清洗 pipeline 输入，三代质量差异仅 0.2% |
| **任务** | 选择合适的输入数据展示 pipeline 效果 |
| **问题** | 在干净数据上跑清洗 pipeline，效果为零——质量提升 0.2%、去重率 0%。Pipeline 无法展示任何价值 |
| **建议** | 换成 CC WET 原始数据（垃圾率~50-70%）。核心原则：清洗 pipeline 需要脏数据输入，评估需要有标签数据，训练需要干净数据 |

### R04. 样本量需从下游倒推

| 维度 | 内容 |
|------|------|
| **场景** | 初始用 1000 条数据，经 Gen1(40%) → Gen2(10%) 后仅剩 40 条 |
| **任务** | 确定合适的输入样本量 |
| **问题** | 40 条数据无统计意义，无法做有效对比分析 |
| **建议** | 从 pipeline 末端往前倒推——最终需要 X 条 → 除以每步保留率 → 得到原始输入量。设计三档体系：smoke_test(12K) / medium_run(36K) / full_run(100K) |

### R05. .gitignore 不支持行内注释

| 维度 | 内容 |
|------|------|
| **场景** | .gitignore 文件中写 `*.bin # model files` |
| **任务** | 正确配置 .gitignore 排除模型文件 |
| **问题** | `#` 被解析为文件名的一部分，导致 491MB 的 .bin 模型文件差点被提交到 git |
| **建议** | .gitignore 行内注释必须单独一行。检查方法：提交前 `git status` 确认无大文件 |

### R06. 多源分层采样避免偏斜

| 维度 | 内容 |
|------|------|
| **场景** | 从 Common Crawl 采样数据 |
| **任务** | 获得有代表性的采样数据 |
| **问题** | 从单一分片顺序读取会导致域名集中、语言偏斜 |
| **建议** | 从 10 个随机 segment 分层采样，合并后全局 shuffle。验证：域名数 ≥ 目标 80%，单域名占比 < 5% |

### R07. CC WET 英文比例远低于预期

| 维度 | 内容 |
|------|------|
| **场景** | 随机 segment 中英文仅~25%（预期 40-50%） |
| **任务** | 理解语言过滤高过滤率的原因 |
| **问题** | 语言过滤成为最大过滤器（75.4%），Gen1 整体保留率仅 3.4%（远低于论文 30-40%） |
| **建议** | 这是 CC WET 数据特征，非 bug。语言过滤后的数据质量正常，Gen1/Gen2/Gen3 质量梯度明显 |

### R08. Jupyter surrogate 字符处理

| 维度 | 内容 |
|------|------|
| **场景** | CC WET 数据在 Notebook 中打印时报 UnicodeEncodeError |
| **任务** | 安全展示 CC WET 原始文本 |
| **问题** | `encode('utf-8', errors='replace')` 不够，surrogate 字符仍会崩溃 |
| **建议** | 使用 `re.sub(r'[\ud800-\udfff]', '', text)` 主动移除 surrogate 字符 |

### R09. 阈值必须对标数据分布

| 维度 | 内容 |
|------|------|
| **场景** | 论文 C4 terminal_punct_min_ratio 阈值 0.7，CC WET 全行 P50 仅 0.09 |
| **任务** | 校准过滤阈值以适应 CC WET 数据 |
| **问题** | 直接复制论文阈值导致过滤率 89%（过严），几乎杀掉所有文档 |
| **建议** | 先测分布（P10/P25/P50/P75/P90），再定阈值。实际修复：只计算内容行（>10词）的 tp_ratio，阈值降至 0.3 |

### R10. 配置值传参 vs 硬编码默认值

| 维度 | 内容 |
|------|------|
| **场景** | YAML config 定义了阈值，但代码构造函数使用硬编码默认值 |
| **任务** | 确保 config 生效 |
| **问题** | 修改 config 完全无效——构造链断裂，参数未从 YAML 传入子组件。Gen1 一直用默认阈值运行 |
| **建议** | 追踪构造链：YAML → config loader → pipeline → 子组件，每层确认参数传递。验证方法：改某阈值为极端值，观察过滤率是否变化 |

## 阶段七：NB 增强与五维 Profiling（2026-03-07）

### R11. Notebook 增加五维数据质量 Profile

| 维度 | 内容 |
|------|------|
| **场景** | NB02-04 只有单一维度的质量评估 |
| **任务** | 全面展示数据清洗效果 |
| **问题** | 单一指标（自训练分类器）存在循环偏差风险 |
| **建议** | 增加五维 Profile：规模（文档数/长度）+ 质量（KenLM PPL）+ 语言（fastText lid）+ 多样性（n-gram ratio）+ 毒性（detoxify）。循环偏差防范：自训练分类器 vs KenLM 两者趋势一致才可信 |

### R12. 子过滤器级别的贡献度分析

| 维度 | 内容 |
|------|------|
| **场景** | NB02 只展示大步骤的汇总过滤率 |
| **任务** | 定位过滤瓶颈 |
| **问题** | 用户问"到底是哪个子规则过滤了最多文档？"无法回答 |
| **建议** | 必须细拆到子指标：如 quality_filter → Gopher(870) / C4(1255) / FineWeb(0) → 各子规则（too_short: 522, low_terminal_punct: 812...）|

### R13. 指标口径必须定义分子分母

| 维度 | 内容 |
|------|------|
| **场景** | 文档/Notebook 中出现"过滤率 20%"、"质量提升 +10%" |
| **任务** | 确保所有百分比/比率可解读、可复现 |
| **问题** | 不知道分子分母是什么。"过滤率 20%"是条件过滤率还是绝对过滤率？"+10%"是绝对差还是相对变化率？ |
| **建议** | 任何百分比必须同时说明分子/分母。区分三类百分比：条件过滤率（该步丢弃/该步输入）、绝对过滤率（该步丢弃/原始总量）、端到端保留率（最终输出/原始总量）|

## 阶段八：分类器修复与方法论深化（2026-03-08 上午）

### R14. 分类器训练数据必须同分布

| 维度 | 内容 |
|------|------|
| **场景** | gen2_classifier.bin 在 03-05 用旧 FineWeb（已清洗）训练，03-06 换成 CC WET 后未重训 |
| **任务** | 分类器在 CC WET 上有效区分文档质量 |
| **问题** | 分数全部压缩在 [0.50, 0.55]，完全无区分能力。分离度 < 0.1 |
| **建议** | 同分布三要素：① 正/负样本来自不同质量层级 ② 文本长度自适应归一化 ③ 训练后 sanity check（分离度 > 0.1）。5 个模型全部删除重训，分离度从 <0.1 提升到 >0.8 |

### R15. 正样本不截断，只截断较长方

| 维度 | 内容 |
|------|------|
| **场景** | 讨论分类器训练时的文本截断策略 |
| **任务** | 消除长度作为伪特征（covariate shift） |
| **问题** | 固定 200 词截断不适用所有数据源组合。Cosmopedia(536 词) vs Gen1 输出(866 词)比值仅 1.6x，不需截断却被截断 |
| **建议** | 三条原则：① 只截断较长方，正样本完整保留 ② 截断阈值=较短方 p90 ③ 推理时也截断（train-inference 一致）。自适应决策：比值 >3x→p90，2x~3x→p95，<2x→不截断 |

### R16. Gen3 负样本必须用原始数据

| 维度 | 内容 |
|------|------|
| **场景** | Gen3 集成分类器用 Gen1 输出做负样本 |
| **任务** | 分类器有足够区分能力 |
| **问题** | Gen1 输出已过滤，与正样本差距太小，分离度仅 0.14~0.44 |
| **建议** | 改用原始 CC WET 做负样本。分离度从 0.24→0.89（edu），0.44→0.94（dclm）。负样本越"脏"，对比信号越强 |

### R17. fastText 二分类必须用 predict(k=2)

| 维度 | 内容 |
|------|------|
| **场景** | fastText 分类器预测 |
| **任务** | 获取准确的概率分数 |
| **问题** | `predict(text)` 默认 k=1，只返回 top-1 标签概率，无法获取对立标签的概率 |
| **建议** | 二分类器必须用 `predict(text, k=2)` 获取双标签概率 |

### R18. ML 模型训练后必须有 sanity check

| 维度 | 内容 |
|------|------|
| **场景** | 分类器训练完成但未验证效果 |
| **任务** | 确认分类器学到了有效信号 |
| **问题** | 分类器退化为随机分类器时，Pipeline 运行无报错、输出非空，但结果无意义 |
| **建议** | 训练后立即检查：分离度 > 0.1、P90-P50 差值 > 0.2、top-10% vs top-50% 阈值差 > 0.1。异常时排查数据/超参/样本选择 |

### R19. Golden Set 验证方法论

| 维度 | 内容 |
|------|------|
| **场景** | 用户明确要求纳入方法论：用已知预期结果的样本验证 pipeline 正确性 |
| **任务** | 建立 pipeline 可观测性 |
| **问题** | Pipeline 运行通过 ≠ 结果正确。过滤器可能失效但保留率恰好在合理范围 |
| **建议** | 三层验证：① 10 条黄金样本（3 好 + 3 坏 + 4 边界）② 阶段断言（每步检查去留）③ 回归检查（代码变更后无退化）|

### R34. 统计差异调查方法论

| 维度 | 内容 |

| **场景** | 用户明确要求记住：terminal_punct_ratio CC WET P50=0.09 vs 论文 0.7 |

### R35. terminal_punct 根因修复

| 维度 | 内容 |
|------|------|
| **场景** | CC WET 全行 P50=0.09（论文 0.7），差距 8x |
| **任务** | 修复 C4 terminal_punct 过滤器 |
| **问题** | 根因：CC WET 72%+ 行 ≤5 词（导航/菜单短行），稀释了 tp_ratio |

# 第二部分：分类总结

## 一、执行纪律（7 条）

> 核心思想：代码存在 ≠ 任务完成。必须执行、验证、报告证据。

| # | 规则 | 来源 |
|---|------|------|
| 1 | 写一步、跑一步、验一步（增量执行） | R01 |
| 2 | 报告完成时附带证据（日志 + 产出 + 指标） | R02 |
| 3 | Notebook 编辑后必须立即执行，禁止连续编辑 | R55 |
| 4 | 执行后刷新 VS Code + 读取展示输出 | R56 |
| 5 | 多任务逐项核销，未核销禁止报完成 | R57 |
| 6 | 阶段验收：列出全部预期产出物逐一确认 | R02（延伸） |
| 7 | 耗时脚本加 caffeinate -i，使用 python3 | R60, R61 |

**专业评价**：这组规则是整个项目最早也最关键的教训。"代码生成≠任务完成"是 AI 辅助开发的典型陷阱——AI 擅长生成代码但容易跳过验证环节。增量执行和证据报告制度有效解决了这个问题。

---

## 二、架构设计（5 条）

> 核心思想：清晰的职责分离 + 数据契约 + 配置驱动。

| # | 规则 | 来源 |
|---|------|------|
| 1 | 三层职责分离：src/（库）+ scripts/（CLI）+ notebooks/（分析） | R41 |
| 2 | Notebook 禁止导入 Pipeline/Trainer/Filter 等执行模块 | R41 |
| 3 | 数据契约：脚本声明产出文件，Notebook 声明并校验依赖 | R42 |
| 4 | Config 驱动规模（run_config.yaml），代码不硬编码数量参数 | R43 |
| 5 | 配置值必须沿构造链传递到子组件，禁止丢弃 | R10 |

**专业评价**：三层职责分离是保证项目可维护性的基石。最初 NB02 直接运行 6 步过滤、NB03 现场训练分类器，重构后所有 Notebook 只读预计算结果，执行时间从分钟级降到秒级，且彻底消除了"Notebook 状态不可复现"的问题。

---

## 三、数据与实验设计（8 条）

> 核心思想：数据源匹配任务目标，样本量倒推，阈值对标分布。

| # | 规则 | 来源 |
|---|------|------|
| 1 | 数据源必须匹配任务目标（清洗需要脏数据） | R03 |
| 2 | 样本量从下游倒推，不拍脑袋 | R04 |
| 3 | 多源分层采样避免偏斜 | R06 |
| 4 | 阈值必须对标数据分布，不能直接复制论文值 | R09 |
| 5 | 统计差异 >2x 时禁止直接下结论，需两步验证 | R34 |
| 6 | 五维数据质量 Profiling（规模/质量/语言/多样性/毒性） | R11 |
| 7 | 子过滤器级别贡献度分析（不能只有大步骤汇总） | R12 |
| 8 | terminal_punct 根因修复范式（计算内容行而非全部行） | R35 |

**专业评价**：R03（数据源匹配）是最大的前期决策失误——用 FineWeb 当 pipeline 输入导致整个项目重做。这说明在实验设计阶段，必须先问"我的数据够脏/够难吗"，而不是"哪个数据好下载"。R09（阈值对标分布）和 R34（统计差异调查方法论）共同构成了"不盲目信任论文参数"的防御体系。

## 四、ML 模型训练（7 条）

> 核心思想：同分布训练、sanity check、自适应截断。

| # | 规则 | 来源 |

**专业评价**：R14 是项目中最严重的 bug——分类器在旧数据上训练，分数全部压缩在 [0.50, 0.55]，但 Pipeline 运行无报错。这暴露了"ML 模型静默失效"的风险：模型不报错但结果无意义。R18（sanity check）是防御此类问题的关键手段，应成为所有 ML pipeline 的标配。
